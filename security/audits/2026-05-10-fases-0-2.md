Fecha: 2026-05-10
Auditor: Agente SECURITY

## Resumen ejecutivo

Se auditaron los artefactos producidos durante las fases 0 (entorno base), 1 (frontend con Web Speech API) y 2 (backend FastAPI base) del proyecto `asistente-noticias`. La superficie de ataque es todavía pequeña: un frontend estático sin peticiones al backend y un backend con únicamente dos endpoints (`GET /health`, `POST /echo`). Se identificaron **6 hallazgos**: 1 de severidad ALTA (CORS abierto a `*` en el punto de entrada real del servidor), 3 de severidad MEDIA (ausencia de rate limiting, ausencia de headers de seguridad HTTP y falta de validación de longitud en el modelo de entrada) y 2 de severidad BAJA/INFO. No se encontraron secretos hardcodeados ni archivos `.env` commiteados. La gestión de secretos sigue correctamente la política definida. Se recomienda cerrar los hallazgos de severidad ALTA y MEDIA antes de conectar los endpoints con servicios externos (Groq, GNews, NewsData).

## Detalle de hallazgos

### SEC-001 — CORS abierto a `*` en el punto de entrada real del servidor

**Descripción:**
El archivo `backend/main.py` — que es el módulo ejecutado directamente durante la fase 2 — configura `allow_origins=["*"]`. Esto permite que cualquier origen arbitrario realice peticiones cross-origin autenticadas al backend, lo que puede ser explotado para ataques CSRF si en el futuro se añaden cookies o cabeceras de autorización.

**Evidencia:**
Archivo: `/home/user/asistente-noticias/backend/main.py`, líneas 10-15
```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],
    allow_methods=["GET", "POST", "OPTIONS"],
    allow_headers=["Content-Type"],
)
```

El módulo `backend/app/main.py` (estructura modular del proyecto) sí lee `ALLOWED_ORIGINS` desde la configuración:
```python
allow_origins=settings.allowed_origins.split(","),
```
Sin embargo, `backend/main.py` (el que fue creado y commiteado en la fase 2 como punto de entrada del servidor) ignora la variable de entorno y usa el wildcard directamente. Esta dualidad implica que, dependiendo de cómo se arranque el servidor (`uvicorn main:app` vs `uvicorn app.main:app`), la configuración activa puede ser insegura.

**Referencia OWASP:** A05:2021 — Security Misconfiguration

**Recomendación:**
1. Eliminar o corregir `backend/main.py` para que use `settings.allowed_origins.split(",")` igual que `backend/app/main.py`, o consolidar en un único punto de entrada.
2. Asegurarse de que `ALLOWED_ORIGINS` en producción contenga únicamente `https://[usuario].github.io`.
3. Documentar cuál de los dos archivos es el punto de entrada canónico.

**Prioridad:** ALTA

### SEC-002 — Falta de validación de longitud máxima en `EchoRequest`

**Descripción:**
El modelo Pydantic `EchoRequest` define los campos `texto` e `idioma` como `str` sin ninguna restricción de longitud. Un atacante puede enviar payloads de decenas de megabytes, agotando memoria o CPU en el servidor. Esto será especialmente crítico cuando `texto` se reenvíe como prompt al LLM (Groq), donde un payload grande puede agotar cuota de tokens o provocar errores costosos.

**Evidencia:**
Archivo: `/home/user/asistente-noticias/backend/main.py`, líneas 23-25
```python
class EchoRequest(BaseModel):
    texto: str
    idioma: str
```
No hay `max_length`, `constr`, ni `Field(max_length=...)`.

**Referencia OWASP:** A03:2021 — Injection / A05:2021 — Security Misconfiguration

**Recomendación:**
Usar `Annotated` con `StringConstraints` o `Field` de Pydantic para imponer límites:
```python
from pydantic import BaseModel, Field

class EchoRequest(BaseModel):
    texto: str = Field(..., max_length=2000)
    idioma: str = Field(..., max_length=10)
```
Añadir también un middleware de límite de tamaño de body HTTP (p.ej. rechazar peticiones superiores a 16 KB antes de parsear JSON).

**Prioridad:** MEDIA

### SEC-003 — Ausencia de headers de seguridad HTTP en el backend

**Descripción:**
El backend FastAPI no añade ninguno de los headers de seguridad estándar definidos en el checklist del proyecto. Estos headers mitigan ataques de tipo clickjacking, MIME sniffing, XSS y downgrade de protocolo. No existe ningún middleware personalizado ni uso de bibliotecas como `secure` o `starlette-exceptionhandlers` que los inyecte.

**Evidencia:**
Revisión de `/home/user/asistente-noticias/backend/main.py` y `/home/user/asistente-noticias/backend/app/main.py`: ninguna referencia a `X-Content-Type-Options`, `X-Frame-Options`, `Content-Security-Policy`, `Strict-Transport-Security` ni `Referrer-Policy`.

Headers ausentes:
- `X-Content-Type-Options: nosniff`
- `X-Frame-Options: DENY`
- `Strict-Transport-Security: max-age=31536000; includeSubDomains` (producción)
- `Referrer-Policy: strict-origin-when-cross-origin`
- `Content-Security-Policy` (si el backend sirve HTML)

**Referencia OWASP:** A05:2021 — Security Misconfiguration

**Recomendación:**
Añadir un middleware Starlette que inyecte los headers en cada respuesta:
```python
from starlette.middleware.base import BaseHTTPMiddleware

class SecurityHeadersMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        response = await call_next(request)
        response.headers["X-Content-Type-Options"] = "nosniff"
        response.headers["X-Frame-Options"] = "DENY"
        response.headers["Referrer-Policy"] = "strict-origin-when-cross-origin"
        return response
```
En producción añadir también `Strict-Transport-Security`.

**Prioridad:** MEDIA

### SEC-004 — Rate limiting no implementado en ningún endpoint

**Descripción:**
La política de rate limiting (`security/policies/rate-limiting.md`) exige límites por IP en todos los endpoints públicos, con un máximo de 20 peticiones/hora en `/ask` y 60 en `/news`. Actualmente ni `backend/main.py` ni `backend/app/main.py` implementan ninguna forma de rate limiting. No existe `slowapi` ni ningún equivalente en `requirements.txt`. Cualquier cliente puede saturar el servicio o agotar las cuotas de APIs externas de pago (Groq, GNews, NewsData) sin coste alguno para el atacante.

**Evidencia:**
Archivo: `/home/user/asistente-noticias/backend/requirements.txt` — sin referencia a `slowapi`, `limits` ni similar.

Archivo: `/home/user/asistente-noticias/backend/main.py` y `backend/app/main.py` — sin decoradores `@limiter.limit(...)` ni middleware de limitación de tasa.

**Referencia OWASP:** A05:2021 — Security Misconfiguration; CWE-770 — Allocation of Resources Without Limits

**Recomendación:**
Instalar `slowapi` y configurar un limitador global. Ejemplo mínimo:
```python
from slowapi import Limiter, _rate_limit_exceeded_handler
from slowapi.util import get_remote_address
from slowapi.errors import RateLimitExceeded

limiter = Limiter(key_func=get_remote_address, default_limits=["100/hour"])
app.state.limiter = limiter
app.add_exception_handler(RateLimitExceeded, _rate_limit_exceeded_handler)
```
Aplicar límites específicos en `/ask` (20/hora) y `/news` (60/hora) tal como indica la política.

**Prioridad:** MEDIA

### SEC-005 — Frontend sin Content-Security-Policy

**Descripción:**
El archivo `index.html` no incluye ninguna directiva `Content-Security-Policy` (ni como meta tag ni servida como header HTTP). Sin CSP, si en fases futuras se inyecta contenido proveniente del LLM o de las noticias en el DOM, un XSS podría ejecutar scripts arbitrarios en el contexto de la aplicación. El riesgo actual es bajo porque el frontend no realiza peticiones al backend todavía, pero debe mitigarse antes de integrar el flujo completo.

**Evidencia:**
Archivo: `/home/user/asistente-noticias/frontend/index.html`, líneas 1-39. No existe ninguna meta `<meta http-equiv="Content-Security-Policy">` ni comentario que indique que el servidor de GitHub Pages la sirve como header.

El archivo `app.js` usa `textContent` y `.value` para mostrar resultados (no `innerHTML`), lo cual es correcto y mitiga XSS directo en el código actual. La ausencia de CSP es un riesgo preventivo para fases posteriores.

**Referencia OWASP:** A03:2021 — Injection (XSS); OWASP CSP Cheat Sheet

**Recomendación:**
Añadir un meta CSP restrictivo en `index.html`:
```html
<meta http-equiv="Content-Security-Policy"
  content="default-src 'self'; script-src 'self'; style-src 'self'; connect-src https://[backend-url];">
```
Actualizar `connect-src` con la URL real del backend en Render al desplegar.

**Prioridad:** BAJA

### SEC-006 — Puerto inconsistente en `.env.example` raíz

**Descripción:**
El archivo `.env.example` en la raíz del proyecto define `ALLOWED_ORIGINS=http://localhost:8000`, mientras que el backend escucha en el puerto por defecto de uvicorn (8000) pero el `.env.example` de `backend/` indica `http://localhost:5500,http://127.0.0.1:5500` (puerto de Live Server para el frontend). Esta inconsistencia puede confundir a nuevos colaboradores y llevar a configurar CORS incorrectamente, con el riesgo de bloquear el frontend o, peor, de usar valores por defecto inseguros.

**Evidencia:**
`/home/user/asistente-noticias/.env.example`, línea 4:
```
ALLOWED_ORIGINS=http://localhost:8000
```
`/home/user/asistente-noticias/backend/.env.example`, línea 7:
```
ALLOWED_ORIGINS=http://localhost:5500,http://127.0.0.1:5500
```

**Referencia:** Política interna `security/policies/cors-policy.md`

**Recomendación:**
Unificar ambos `.env.example` para que reflejen el mismo valor canónico de desarrollo. El valor correcto es el del `backend/.env.example` (`http://localhost:5500,http://127.0.0.1:5500`), ya que `ALLOWED_ORIGINS` controla qué orígenes del **frontend** puede llamar al backend, no el puerto del propio backend. Actualizar el `.env.example` raíz o eliminarlo si es redundante.

**Prioridad:** BAJA (INFO)

## Hallazgos positivos (sin acción requerida)

- **Secretos no expuestos:** No se encontraron API keys ni contraseñas hardcodeadas en ningún archivo `.py` ni `.js`.
- **`.env` no commiteado:** El archivo `.env` está correctamente excluido por `.gitignore` (línea 151 y línea 229) y no aparece en el historial de git.
- **`.env.example` sin valores reales:** Ambos `.env.example` contienen solo nombres de variables con valores vacíos o de desarrollo, sin secretos reales.
- **`__pycache__`, `.venv`, `node_modules`:** Todos excluidos por `.gitignore`.
- **Dependencias fijadas con `==`:** `requirements.txt` usa versiones exactas (`fastapi==0.115.6`, `uvicorn[standard]==0.32.1`, etc.), eliminando el riesgo de supply chain por versiones flotantes.
- **Docs Swagger deshabilitados en producción:** `backend/app/main.py` configura `docs_url` como `None` cuando `env != "development"`, evitando exponer la documentación interactiva.
- **Manejo de errores de micrófono en frontend:** `app.js` gestiona explícitamente `not-allowed`, `no-speech`, `audio-capture` y `network` sin exponer información técnica interna al usuario.
- **Sin `innerHTML` en el frontend:** El código JS usa `textContent` y `.value` para asignar contenido, evitando XSS directo en el estado actual.
